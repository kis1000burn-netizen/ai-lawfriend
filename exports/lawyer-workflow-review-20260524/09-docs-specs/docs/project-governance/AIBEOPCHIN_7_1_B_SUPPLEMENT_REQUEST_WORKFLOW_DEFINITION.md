# AI법친 7.1-B — 보완 요청 워크플로우 정의서

## 1. 문서 목적

AI법친 7.1-B 보완 요청 워크플로우 정의서는 변호사가 6.x 사건 패키지를 검토한 뒤 의뢰인에게 추가 자료, 추가 답변, 누락 정보, 문서 보완을 요청하고, 의뢰인이 이를 다시 입력·제출한 뒤 변호사가 재검토하는 흐름을 정의하기 위한 기준문서다.

이 문서는 코드 구현 문서가 아니다.  
이 문서는 보완 요청 기능의 주체, 대상, 유형, 상태 흐름, 화면/API/DB 분리 기준을 확정하기 위한 문서다.

## 2. 현재 전제

AI법친 6.x 사건 패키지 기능군은 운영 배포 및 Smoke Test 14/14 PASS 이후 최종 잠금 완료 상태다.

AI법친 7.1의 1차 확정 후보는 "보완 요청 워크플로우"다.

현재 전제:

- 6.x 운영 배포 완료
- 6.x Smoke Test 14/14 PASS
- 6.x 최종 잠금 완료
- 6.x 신규 기능 추가 금지
- 7.x는 6.x와 별도 로드맵으로 분리
- 7.1 후보 확정 정의서 완료
- 7.1-B는 6.x 사건 패키지 공유 정책을 직접 변경하지 않는다
- 7.1-B는 보완 요청 흐름을 별도 워크플로우로 정의한다

## 3. 핵심 개념

보완 요청은 변호사가 사건 패키지 검토 중 발견한 누락 정보, 불명확한 사실관계, 추가 자료 필요사항, 문서 보완 필요사항을 의뢰인에게 요청하는 절차다.

보완 요청은 법률 판단 확정이 아니다.  
보완 요청은 사건 패키지의 완성도를 높이기 위한 정보 보강 절차다.

## 4. 보완 요청 주체

### 4.1 요청 생성 주체

보완 요청을 생성할 수 있는 주체는 다음으로 제한한다.

| 주체 | 가능 여부 | 설명 |
|---|---:|---|
| 변호사 | 가능 | 본인이 열람 권한을 가진 사건 패키지에 대해 보완 요청 가능 |
| 관리자 | 제한 가능 | 운영상 필요한 경우 대리 기록 또는 상태 확인 가능. 단, 법률 검토 내용 직접 작성은 원칙적으로 변호사 주체 |
| 의뢰인 | 불가 | 의뢰인은 보완 요청을 생성하지 않고 응답만 가능 |
| AI | 불가 | AI는 후보 문안 추천까지만 가능하며 보완 요청 확정 생성 주체가 될 수 없음 |

### 4.2 요청 응답 주체

| 주체 | 가능 여부 | 설명 |
|---|---:|---|
| 의뢰인 | 가능 | 본인 사건에 대한 보완 요청에 응답 가능 |
| 변호사 | 가능 | 응답 내용 검토 및 재요청 가능 |
| 관리자 | 제한 가능 | 운영상 상태 확인 가능 |
| AI | 제한 | 응답 정리·요약 보조 가능. 최종 판단 불가 |

## 5. 보완 요청 대상

보완 요청은 다음 대상에 연결될 수 있다.

| 대상 | 설명 |
|---|---|
| 사건 패키지 전체 | 사건 전체에 대한 보완 요청 |
| 사건 요약 | 사건 개요, 타임라인, 피해 내용 등 요약 정보 보완 |
| 인터뷰 답변 | 기존 AI 인터뷰 답변 중 누락·불명확 항목 보완 |
| 첨부자료 | 계약서, 문자, 계좌이체 내역, 사진, 녹취 등 증거자료 추가 요청 |
| 문서 초안 기초 | 진술서, 사실관계 정리서, 상담 전 질문 목록 등 문서 초안 기초 보완 |
| 특정 쟁점 후보 | 핵심 쟁점 후보에 대한 추가 사실 확인 |
| 개인정보/당사자 정보 | 이름, 연락처, 주소, 상대방 정보 등 기본 정보 보완 |

## 6. 보완 요청 유형

보완 요청 유형은 다음으로 정의한다.

| 유형 코드 | 유형명 | 설명 |
|---|---|---|
| MISSING_FACT | 누락 사실 보완 | 사건 경위 중 빠진 사실 보완 |
| UNCLEAR_FACT | 불명확 사실 확인 | 기존 답변이 모호하거나 상충될 때 확인 |
| ADDITIONAL_EVIDENCE | 추가 증거자료 요청 | 첨부자료 추가 제출 요청 |
| DOCUMENT_CLARIFICATION | 문서 내용 보완 | 문서 초안 기초 또는 요약 내용 보완 |
| PARTY_INFO | 당사자 정보 보완 | 의뢰인, 상대방, 참고인 정보 보완 |
| TIMELINE_CONFIRMATION | 시점/순서 확인 | 사건 발생일, 지급일, 통화일 등 날짜·순서 확인 |
| DAMAGE_DETAIL | 피해금액/손해내용 보완 | 금액, 손해, 변제 여부 등 보완 |
| CONSENT_OR_NOTICE | 동의/고지 확인 | 공유, 개인정보, 열람 관련 안내 확인 |
| OTHER | 기타 보완 | 위 유형에 속하지 않는 기타 보완 요청 |

## 7. 보완 요청 상태값 초안

상태값은 별도 상태값 정의서에서 최종 확정한다.  
이 정의서에서는 워크플로우 이해를 위한 초안만 둔다.

| 상태 | 의미 |
|---|---|
| DRAFT | 변호사가 작성 중 |
| SENT | 의뢰인에게 발송됨 |
| CLIENT_VIEWED | 의뢰인이 확인함 |
| CLIENT_RESPONDED | 의뢰인이 응답함 |
| UNDER_REVIEW | 변호사가 응답 내용을 재검토 중 |
| NEEDS_MORE_INFO | 추가 보완이 다시 필요함 |
| ACCEPTED | 보완 내용이 수용됨 |
| CLOSED | 보완 요청 흐름 종료 |
| CANCELLED | 요청 취소 |
| EXPIRED | 응답 기한 만료 |

상태값 canonical source는 이후 `AIBEOPCHIN_7_1_B_SUPPLEMENT_REQUEST_STATUS_DEFINITION.md`에서 확정한다.

## 8. 기본 워크플로우

### 8.1 정상 흐름

```text
변호사 사건 패키지 검토
→ 보완 요청 작성
→ 의뢰인에게 보완 요청 발송
→ 의뢰인 보완 요청 확인
→ 의뢰인 답변/자료 제출
→ 변호사 재검토
→ 수용 또는 추가 보완 요청
→ 보완 요청 종료
```

### 8.2 재요청 흐름

```text
의뢰인 응답
→ 변호사 재검토
→ 응답 불충분 판단
→ NEEDS_MORE_INFO
→ 추가 보완 요청
→ 의뢰인 재응답
→ 재검토
```

### 8.3 취소 흐름

```text
DRAFT 또는 SENT
→ 변호사 또는 운영자 취소
→ CANCELLED
```

### 8.4 만료 흐름

```text
SENT 또는 CLIENT_VIEWED
→ 응답 기한 경과
→ EXPIRED
```

## 9. 의뢰인 재입력 흐름

의뢰인은 보완 요청을 받으면 다음 방식으로 응답한다.

1. 보완 요청 목록 확인
2. 요청 유형과 설명 확인
3. 필요한 답변 입력
4. 필요한 첨부자료 업로드
5. 제출 전 고지문 확인
6. 보완 응답 제출
7. 제출 완료 상태 확인

의뢰인 화면에서는 다음을 명확히 보여야 한다.

- 요청한 변호사
- 요청 일시
- 응답 기한
- 보완 요청 유형
- 요청 내용
- 필요한 자료
- 제출 전 고지문
- 제출 상태
- 재요청 여부

## 10. 변호사 재검토 흐름

변호사는 의뢰인 응답 후 다음을 확인한다.

1. 보완 응답 목록 확인
2. 응답 내용 확인
3. 첨부자료 확인
4. 기존 사건 요약과 비교
5. 추가 보완 필요 여부 판단
6. 수용 또는 재요청 처리
7. 필요 시 사건 패키지 검토 메모와 연결

변호사 화면에서는 다음을 명확히 보여야 한다.

- 요청별 응답 상태
- 의뢰인 제출 내용
- 제출 첨부자료
- 응답 일시
- 재요청 이력
- 수용 여부
- 내부 검토 메모 연결 후보

## 11. 6.x 사건 패키지와의 연결 방식

7.1-B는 6.x 사건 패키지 기능을 직접 수정하지 않는다.

다만 다음 항목과 참조 관계를 가질 수 있다.

| 6.x 대상 | 연결 방식 |
|---|---|
| Case | 보완 요청의 기준 사건 |
| CasePackageShare | 변호사 열람 권한 확인 기준 |
| CasePackageAccessLog | 보완 요청 생성 전후 열람 이력 참고 |
| CaseAttachment | 추가 자료 요청 및 응답 첨부 연결 후보 |
| CaseSummary | 보완 후 요약 갱신 후보 |
| LegalDocument | 문서 초안 기초 보완 후보 |
| AuditLog | 보완 요청 생성/응답/수용/취소 기록 후보 |

중요 원칙:

- 6.x 공유 정책을 직접 변경하지 않는다.
- 6.x 고유번호 정책을 변경하지 않는다.
- 6.x 첨부파일 다운로드 정책을 완화하지 않는다.
- 6.x PDF 엔진을 즉시 변경하지 않는다.
- 7.1-B는 별도 데이터/상태/API 정의 이후 구현한다.

## 13. 고지문 원칙

보완 요청 화면에는 다음 고지 원칙이 적용되어야 한다.

- 보완 요청은 변호사의 검토를 위한 추가 정보 요청이다.
- AI가 법률판단을 확정하지 않는다.
- 의뢰인이 제출한 정보는 사건 검토 목적에 사용된다.
- 민감자료 제출 전 개인정보 포함 여부를 확인해야 한다.
- 제출 자료는 열람/다운로드 권한 정책에 따라 관리된다.
- 최종 법률 판단과 문서 사용 여부는 변호사 검토를 거친다.

## 18. QA 기준 초안

향후 QA 체크리스트에서 다음 항목을 검증한다.

- 변호사만 보완 요청 생성 가능
