# 공부호 표준 (GONGBUHO_STANDARD)

**상태**: Phase 1 초안 · 구현(DB/API) 전제와 독립적으로 합의·수정 가능

## 1. 공부호 정의

**공부호(Gongbuho)**는 특정 서비스 도메인(예: AI법친)에서, **사건 유형(또는 사건 카테고리)**에 대해 다음을 한 패킷으로 묶은 **버전 관리 가능한 사고 구조 정의**이다.

- 질문 흐름(인터뷰·증빙 수집)
- 추론·정리 순서(요약·쟁점 후보 생성 시 골격)
- 검증 규칙(출력물 품질·금지 패턴 방지 보조)
- 출력 계약(요약 섹션·문서 종류 목록 등)
- 금지 규칙(확정적 법률 판단·위험 유도 차단 등)
- 전문가(변호사) 검토 포인트 표시 정보
- 인간 확인·승인에 관한 플래그

공부호는 **챗봇 성격 기능이 아니라**, 기존 “사건 → 질문셋 → 인터뷰 → 요약 → 문서 → 변호사 검토” 흐름 위에서 **품질·일관성·추적**을 제공하는 상위 레이어다.

동일 패킷 명세는 **AI법친 외**(예: 향후 CMB Studio 등) 다른 제품 라인에서도 재사용할 수 있도록, AI법친 전용 문자열 의존만 `domain` 또는 `tenant` 같은 필드로 격리한다.

## 2. 패킷 식별

| 필드 | 필수 | 설명 |
|------|------|------|
| `code` | 예 | 사람이 읽는 고유 코드. 예: `LAW-FRAUD-001` |
| `name` | 예 | 패킷 표시명 |
| `version` | 예 | **[SemVer 2.0.0]** 권장. 동일 `code`에 대해 복수 버전 공존 가능 |
| `domain` | 예 | 패킷 적용 소속. 예: `AI법친` |
| `caseType` | 예 | 사건 매칭 키(앱 코드와 매핑). 예: `FRAUD` |
| `locale` | 아니오 | 예: `ko-KR`(기본 `ko-KR` 가정 가능) |

`code` + `version` 조합으로 **패킷 인스턴스**를 특정한다. DB 저장 시에는 별도 `id`(cuid 등)가 있어도 됨.

## 3. 상태(승인) — 저장소(Persistence) 레벨

패킷이 DB에 존재할 때 권장하는 수명 상태(구현 세부는 Prisma 등에서 이름 조정 가능).

| 상태 | 의미 |
|------|------|
| `DRAFT` | 편집 중, 사용자 사건 적용 불가 또는 경고 필요 |
| `REVIEW` | 내부 검토 단계 |
| `APPROVED` | **사건에 적용 가능한 유일 안정 상태**(프로덕션 매칭 기본값) |
| `ARCHIVED` | 더 이상 신규 적용 불가·이력만 유지 |

**원칙**: 프로덕션에서 사건에 자동 매칭되는 패킷은 **`APPROVED`만**.

## 4. 패킷 JSON 구조 — 필수 / 선택 / 금지

구현에서는 JSON을 그대로 `Json` 칼럼에 넣거나, 큰 블록을 분할 저장할 수 있다. **논리적 표준**은 아래이다.

**캐논 루트 키(이름 계약)**  
샘플 [samples/LAW_FRAUD_001_GONGBUHO.json](./samples/LAW_FRAUD_001_GONGBUHO.json)과 동일한 **camelCase 키**를 사용한다. Prisma 저장·REST 응답·프론트 상태에서 **동일 문자열을 재사용**한다. 세부 규격은 **[GONGBUHO_FIELD_MAPPING.md](./GONGBUHO_FIELD_MAPPING.md)** 참조.

루트 키 이름(등장 순서 권장, 설명 참고만):

`code`, `name`, `domain`, `caseType`, `version`, `locale`(선택), `meta`(선택), `knowledgeMap`(선택·권장), `reasoningFlow`, `questionFlow`, `validationRules`, `outputContract`, `forbiddenRules`, `expertReviewPoints`, `humanApproval`

### 4.1 필수 블록(논리)

- `questionFlow`: 질문·단계 목록(인터뷰 골격)
- `reasoningFlow`: 사건 정리·요약 순서(문구가 아니라 **단계 키** 배열로도 가능)
- `validationRules`: 생성·요약 과정 검증 규칙 목록
- `outputContract`: 요약·문서에 대한 **출력 골격**
- `forbiddenRules`: 반드시 시스템·프롬프트 레벨에서 반영해야 할 금지 사항

### 4.2 강하게 권장

- `knowledgeMap` 또는 이에 준하는 구조: 핵심 구성요소·증거 유형 목록(인터뷰 라우팅·UI 안내 보조)
- `expertReviewPoints`: 변호사 화면에 노출할 검토 후보 목록 템플릿
- `humanApproval`: 의뢰인 확인·변호사 검토·자동 제출 불가 여부 등

### 4.3 금지되는 패킷 수준 행위(정의서 관점)

다음만으로 패킷 표준을 깨거나 운영 정책과 충돌하면 **승인 불가**로 간주한다.

- **확정적 법률 판단**을 허용하거나 장려하는 규약
- 허위·조작 유도 조항
- 수사 회피·위법 행위 시나리오의 정당화 템플릿
- “변호사 없이 최종 소송 서면·수사 자료로 제출”을 정상 허용하는 것처럼 읽히는 기본 플래그

자세한 런타임 차단 규격은 **[GONGBUHO_SAFETY_POLICY.md](./GONGBUHO_SAFETY_POLICY.md)**.

## 5. 버전 관리 원칙

1. **`APPROVED` 패킷은 불변에 가깝게 취급**: 수정이 필요하면 **새 `version`(또는 새 행)**을 만든다.
2. 변경 시 **changelog** 레벨의 메타(변경 요약·적용 시작일 권장)를 남긴다.
3. 사건 적용 시점에는 **`code`·`appliedVersion`(또는 패킷 `id`)**를 Trace에 동결 기록한다.

## 6. GongbuhoTrace(적용 추적)

**Trace**는 “패킷 정의 파일”이 아니라 **특정 `Case`(또는 동등 엔티티)에 패킷이 적용된 사실**을 남기는 스냅샷이다.

**원칙**

- 패킷이 바뀌어도 **과거 Trace는 수정하지 않는다**(추가만).
- 포함 권장: 적용 패킷 식별자, 버전, 입력 스냅샷 참조 또는 해시, 검증 결과, 위험 플래그, 인간 확인 상태, 타임스탬프.

**목적**: 감사, AI 신뢰·고지 근거, 분쟁 시 “어떤 기준으로 생성되었는가” 증명.

## 7. 인간 승인(Human-in-the-loop)

패킷 `humanApproval` 및 서비스 전역 정책과 합치되게 다음을 둔다.

- 고객에게는 **결론·최종 문서 책임**이 AI에 있지 않음을 고지
- 제출물·패키지 공유·외부 로 전달 가능한 형태에서는 **변호사(또는 정책상 지정 역할)** 확인 단계

정책 문구는 안전 문서 참조.

## 8. 전문가 검토 원칙

- AI 출력은 **“검토 큐 후보”**로만 존재할 수 있다.
- `expertReviewPoints`는 가능한 한 **증거 부족·경계 법적 쟁점·금액 산정·당사자 동일성**처럼 **구체 카테고리**를 가진다.
- 변호사 UI에는 “AI가 놓친 게 아니라, **먼저 보라고 표시한 것**”으로 표현 가능한 카피를 권장.
